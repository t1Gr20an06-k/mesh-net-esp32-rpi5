# Полевые тесты

## Цели тестирования

1. Реальная дальность LoRa в горном рельефе (прямая видимость vs. склон)
2. Надёжность доставки SOS через 2 ретранслятора
3. Работа captive portal на iOS/Android без доп. настроек *(этап 5)*
4. Корректность GPS-координат с телефона через HTTPS на ESP32
5. Двусторонний чат под нагрузкой (collision при синхронных нажатиях)

---

## Протокол тестирования

### Тест дальности (FT-01)

**Оборудование:** 2 × RPi5 + SX1262, ноутбук с логгером

**Процедура:**
1. Фиксированная точка (инфо-точка / база) — на возвышении
2. Мобильная точка — удаляется по маршруту с шагом 500 м
3. На каждой точке отправить 10 пакетов PING, записать RSSI/SNR/доставку
4. Зафиксировать GPS-координаты каждой точки тестирования

**Таблица результатов:**

| Точка | Расстояние | Высота разница | RSSI avg | SNR avg | Доставлено / 10 |
|-------|-----------|----------------|----------|---------|-----------------|
| T1 | 500 м | +50 м | | | |
| T2 | 1000 м | +100 м | | | |
| T3 | 2000 м | +200 м | | | |
| T4 | 3000 м | — | | | |
| T5 | 5000 м | — | | | |

**Ожидаемый результат:** RSSI > −130 дБм, доставка > 8/10 на дистанции до 3 км.

### Тест SOS через ретрансляторы (FT-02)

**Схема:** Терминал → Инфо-точка → База спасателей

**Процедура:**
1. Все 3 узла вне прямой видимости друг друга
2. Нажать кнопку SOS на терминале
3. Зафиксировать время: T_send → T_receive (на базе)
4. Повторить 20 раз

**Метрики:**
- Процент доставки (цель: 100%)
- Время доставки P50, P95 (цель: < 5 сек)
- Количество дубликатов (цель: не более 3 копий = ровно SOS_REPEAT)

### Тест автономности терминала (FT-03)

**Оборудование:** ESP32 + SX1262 + GPS(с телефона) + 18650 3500 мАч

**Режим:** PING каждые 60 с, дисплей включён, Wi-Fi AP активен

**Измерения:** Ток потребления (мА), время работы до разряда

**Ожидаемый результат:** > 24 часов

### Тест чата турист↔база (FT-05)

**Оборудование:** ESP32 + смартфон + RPi5 базы + ноутбук с дашбордом

**Сценарий A — последовательный обмен (должно работать всегда):**
1. Оператор пишет «Где вы?» → отправить
2. Турист видит сообщение, отвечает «У ручья» → отправить
3. Оператор видит ответ, пишет «Жду через час» → отправить
4. Турист видит, отвечает «Понял»

Метрики:
- Доставка 4/4 сообщений
- Задержка от нажатия «отправить» до появления у получателя — < 5 сек

**Сценарий B — синхронные нажатия (известное ограничение):**
1. Оператор и турист одновременно (разница ≤ 200 мс) пишут друг другу
2. Зафиксировать: дошло обоим / только одному / никому
3. Повторить 10 раз с разной задержкой между нажатиями (0, 100, 300, 500 мс)

Ожидаемое (с текущим CSMA/3-копий retransmit):
- Разница 0–200 мс: 50–80% потеря (ограничение half-duplex без ACK)
- Разница > 500 мс: 0% потеря — pre-CAD jitter и LBT успевают развести

**Сценарий C — длинный чат во время SOS-бёрста:**
1. Турист нажимает SOS-кнопку (бёрст 3×500 мс)
2. Оператор сразу пишет «Принял, помощь идёт» → отправить
3. Турист после получения SOS-подтверждения отвечает «Жду»

Должно: SOS приходит, ack/resolve работают, чат не застревает.

**Логи для разбора:**
- ESP32: `pio device monitor -b 115200` — `[TX] CHAT`, `[RX] type=1 dev=1`
- База: `sudo journalctl -u mesh-lora-station -f` —
  `[OUTBOX]`, `[LBT] N попыток`, `[RX#N] CHAT dev=16`

## Известные ограничения

- LoRa SF10/BW125 даёт ~980 бит/с — достаточно для координат и коротких
  сообщений, но не для передачи изображений
- CHAT-payload жёстко ограничен **48 байтами UTF-8** (≈24 русских символа).
  Длиннее — обрезается на стороне отправителя
- В глубоких ущельях дальность значительно снижается
- Captive portal не открывается автоматически при использовании VPN на
  смартфоне *(этап 5)*
- **Half-duplex collision при синхронной TX обоих узлов** (см.
  [`CLAUDE.md`](../CLAUDE.md) → «Известные ограничения»). Mitigation —
  CSMA/CAD + 3 копии retransmit. Полное решение требует ACK-протокола,
  отложено до полевых испытаний
